MedPlus, Apollo, Netmeds, and Medkart do not ship a single public, keyless JSON “search price” API you can call from PaxMed without a commercial / technical onboarding (base URL, auth, and schema they give you).

Tata 1mg is the one with published merchant documentation (Onedoc): TATA 1mg merchant API docs — still not usable without the credentials and endpoints they issue after signup.

So “integrate sanctioned API” here means: real fetch() to URLs and auth you configure in .env, not hard-coded secrets or scraping.

What was implemented
server/integrations/partners/partnerEnv.js
Maps each UI retailer to env prefixes: MEDPLUS, APOLLO, NETMEDS, ONE_MG, MEDKART and reads:

{PREFIX}_PARTNER_API_BASE (required for live calls)
{PREFIX}_PARTNER_BEARER_TOKEN or {PREFIX}_PARTNER_API_KEY (+ optional header name)
Optional: SEARCH_PATH, SEARCH_METHOD, QUERY_PARAM, POST_BODY_TEMPLATE, EXTRA_HEADERS_JSON, timeout
server/integrations/partners/fetchPartnerSearch.js
Performs authenticated GET (default) or POST JSON calls to your partner base URL.

server/integrations/partners/parseOfferJson.js
Best-effort parsing of common field names (selling_price, price, mrp, etc.). If a partner’s JSON differs, adjust this or add a per-partner parser once you have their schema.

server/integrations/onlinePharmacies.js → quoteProvider()

If {PREFIX}_PARTNER_API_BASE is set → real HTTP + parse → data_mode: "partner_api".
If not set → ok: false, data_mode: "unconfigured" (no made-up price).
Optional local demo only: ONLINE_USE_ILLUSTRATIVE_FALLBACK=true → old deterministic numbers, data_mode: "illustrative_fallback".
.env.example, README (partner section + 1mg Onedoc link), UI copy on the online panel, and a clearer status line in public/app.js when nothing is configured.

Here’s what was added for Apollo and Netmeds using the same pattern as MedPlus: optional consumer search when B2B *_PARTNER_API_BASE is not set.

Apollo (server/integrations/apolloCatalog.js)
GET https://search.apollo247.com/v4/search?query=…&pincode=…
Headers: authorization (raw value from DevTools — not forced to Bearer), x-app-os: web, x-source-service: PHARMA_AP_IN, x-token: (empty), x-unique-session-id (new UUID per request), Referer like the site.
Env: APOLLO_CATALOG_AUTHORIZATION (required), APOLLO_CATALOG_PINCODE (optional, default 400001 so you never send pincode=undefined).
Parses data.productDetails.products[]: specialPrice → selling price, price → MRP when present.
data_mode: apollo_catalog / apollo_catalog_error
Netmeds (server/integrations/netmedsCatalog.js)
GET https://www.netmeds.com/ext/search/application/api/v1.0/products?page_id=*&page_size=…&q=…
Headers: Authorization: Bearer …, x-currency-code: INR, x-location-detail (JSON string), Referer to /products/?q=…
Env: NETMEDS_CATALOG_BEARER or NETMEDS_CATALOG_AUTHORIZATION (full value; Bearer added if missing). Optional NETMEDS_CATALOG_LOCATION_JSON; otherwise defaults to 110001 / Delhi (same idea as your sample).
Parses items[]: price.effective.min, price.marked.min
data_mode: netmeds_catalog / netmeds_catalog_error
Note: Your browser also sends x-fp-signature and cookies. Many calls still work with only Bearer + location; if Netmeds starts failing, refresh tokens/headers from DevTools or use contract NETMEDS_PARTNER_*.

Product read (market lens)
PaxMed sits at the intersection of price transparency, local pharmacy discovery, online retailer comparison, and diagnostics / booking-adjacent flows—with home delivery orders, reminders, and WhatsApp hooks. That is a credible wedge in India, where users routinely compare MRP vs discount, check availability by area, and juggle apps (1mg, Apollo, Netmeds, Practo, local chemist WhatsApp).

Below is a PM-style gap analysis: what the market expects vs where the product is thin or missing.

What the market rewards (and incumbents already do well)
Trust: licensed sellers, clear “who fulfils”, refund/cancellation policy, prescription handling, data privacy.
Completeness: stock truth, pack size, generic substitutes, delivery ETA bands, COD vs prepaid.
Habit: reorder, subscriptions, offers, wallet/cashback, push notifications.
Diagnostics: slot booking, phlebotomist tracking, report storage, refunds for no-show.
B2B / supply: pharmacy onboarding, catalog sync, margin tools (you have ERP import—market wants live sync and reconciliation).
Strategic gaps (prioritized)
1. Trust & compliance (P0 in India)

Prescription capture, storage policy, Schedule H/X handling, age-restricted meds, and explicit pharmacy licence display per offer are table stakes for serious consumer traction.
Gap: Positioning as “demo / compare” is fine for dev; for market you need a trust layer (seller verification, T&C, grievance, DPDP-aware consent).
2. “Source of truth” for price & stock

Aggregators win on freshness and coverage. Demo datasets and partial partner feeds risk wrong prices—users blame the product, not the data.
Gap: SLA on price freshness, out-of-stock signals, and confidence labels (“verified 2h ago” vs “indicative”). Consider user-reported price or pharmacy-confirmed feed.
3. Conversion path after compare

You deep-link to retailers and support PaxMed delivery for local—good. Market expects one-tap checkout or saved addresses + payment everywhere.
Gap: Unified checkout only where you control fulfilment; elsewhere, track handoff (did user complete purchase?) and recover abandoned compare sessions.
4. Retention & LTV

One-off compare has low LTV. Incumbents own refills, diagnostics bundles, insurance, and chronic care.
Gap: Chronic condition journeys (diabetes, BP, thyroid) with scheduled labs + medicine + reminders in one narrative—not just isolated reminders.
5. Diagnostics as a full product

Practo/Healthians/Orange set expectations: slots, partner SLAs, cancellations, report vault.
Gap: Post-booking status surface (collection done, report ready), report upload/view, and dispute/refund flows. Partner enable/disable is fine; users need predictable outcomes.
6. Geography & discovery

India is PIN-first and hyperlocal.
Gap: PIN-level default, “pharmacies near me” ranking, and coverage maps (“we have 12 stores in this PIN”). You have pieces; market wants confidence before travel.
7. Monetization clarity

Market assumes commission on redirect, listing fees for pharmacies, lead fees, or B2B SaaS.
Gap: A clear business model shapes UX (sponsored ranking vs neutral sort). Neutrality is a brand asset but needs transparent ranking rules.
8. B2B pharmacy story

Marg/RetailGraph import is a strong India-specific hook.
Gap: Partner portal (not just API keys): dashboards, SKU mapping, price push, order inbox from PaxMed deliveries, settlements. Your service_provider order events are a start—market wants operational workflow.
9. Mobile & notifications

Flutter exists; market is app-first + WhatsApp.
Gap: Push for order/reminder, deep links from WhatsApp to order tracking, and offline-friendly compare history on device.
10. Content & SEO

Medicine pages drive organic traffic (1mg-style content).
Gap: SEO landing pages per molecule + city, Hindi/regional copy, and doctor-friendly printable compare summaries (B2B2C).
11. Insurance & payments

UPI is universal; insurance cashless and HSA/ corporate are differentiation at scale.
Gap: Even a “insurance coming soon” roadmap item matters for enterprise buyers.
12. Competitive differentiation (sharp positioning)

“Compare everyone” is crowded. A sharper story: “Verified local + online in one PIN, with honest stock” or “ERP-connected indie pharmacies vs big apps.”
Gap: One sentence that explains why to open PaxMed instead of three retailer apps—then build features that only that story can claim.

. Drug mapping engine
What you have today

medicines as the internal master: display_name, generic_name, strength, form, pack_size, generated search_vector + GIN trigram index.
Search / compare: text match on display_name / generic_name (including “compact” stripping spaces for typos).
Prescription OCR → drug: matchMedicinesFromText uses pg_trgm similarity on search_vector with score threshold and dedupe by medicine_id.
Imports: Xlsx long-format creates/links rows; SKU-style values can land in generic_name (e.g. import:sku:… in seeded data), which is a light form of retailer SKU → row but not a clean mapping model.
Retailers (online): partner search is query-in → offer-out; no shared PaxMed product id across channels in that path.
Gaps vs a real “mapping engine”

No canonical molecule + strength + form + pack layer (duplicates and near-duplicates are expected at scale).
No external vocab (INN, RxNorm, proprietary national codes) as optional references.
No dedicated alias / synonym / brand → generic graph.
Online partner offers are not resolved to one internal medicine_id (yet).
Suggested approach

Phase A (data model): Introduce product_concept (or drug_master): canonical salt, strength, unit, form; migrate medicines to point at it or add master_id + keep medicine as “SKU / pack variant”.
Phase B (mapping): Tables: external_codes (source, code, concept_id), medicine_aliases (text → concept_id, confidence). Keep trigram + rules for fuzzy match; add human review queue for low confidence.
Phase C (pipelines): On import and on partner API response, run resolver → medicine_id / concept_id; store match confidence on pharmacy_prices or staging tables.
Phase D (quality): Periodic dedupe jobs and “merge medicine” admin flow.
2. Price normalization
What you have today

price_inr, mrp_inr, discount_pct on pharmacy_prices (and labs); import can derive selling price from MRP + discount; warnings on discount vs MRP mismatch.
Compare surfaces min/max and spread %; geo sort is separate from “unit price”.
Online: parseOfferJson-style extraction; amounts are whatever the partner sends (no universal per-unit normalization in the compare API).
Gaps

Heterogeneous packs (10s vs 15s) are not normalized to price per unit of therapy (per tablet, per ml, per day) for fair ranking.
No explicit channel normalization (retail vs online vs “offer” types) beyond price_type.
Suggested approach

Define normalized_unit_price_inr (or compute at read time): e.g. price_inr / (pack_size * countable_units) with rules per form.
Enforce pack_size and UOM on import; reject or flag ambiguous rows.
For online partners, map SKUs to medicine_id first, then normalize with the same function as DB prices.
Optionally store price history (pharmacy_price_events) for trending, not just updated_at.
3. Availability intelligence
What you have today

pharmacy_prices.in_stock (boolean) + updated_at.
Diagnostics: partner serviceability paths; city/pin scoped catalog.
Consumer UX: geo distance, sort; no stock confidence in ranking.
Gaps

No quantity, restock ETA, or source of stock (POS vs manual).
No staleness signal (e.g. “price 30 days old”).
No aggregation like “in stock in X% of pharmacies in PIN”.
Suggested approach

Extend model: stock_qty optional, stock_status enum (in_stock / limited / on_request / unknown), stock_observed_at.
Ingest from POS/feeds where possible; default UI to confidence labels (“Verified today” vs “Listed, not verified”).
Rankers: boost in-stock + fresh; penalize unknown when user needs delivery today.
Analytics (see below): track out-of-stock after click to improve partner data.
4. B2B analytics
What you have today

Partner API (requirePartner): /partner/sales/summary, /partner/sales/recent — KPIs, top SKUs, daily series from sales / sale_items (-margin style with unit_cost_inr).
Import + partner dashboard (partner.html).
Operational: Razorpay/webhook and order flows; not the same as marketplace analytics.
Gaps

Little platform-side B2B intelligence: search volume, compare impressions, conversion (search → cart → order), benchmarking vs peers (anonymized), category heatmaps.
Partner analytics depend on sales being fed; ordering via PaxMed may not fully mirror that table unless integrated.
Suggested approach

Event log (append-only): search_query, medicine_id, pharmacy_id, event (impression, click, add_to_cart, checkout_start, order), timestamps, optional anonymized session.
Partner console v2: “Demand for your SKUs”, “Win rate vs cheapest in city”, “Stale prices”.
Chain / admin: rollup across pharmacies (with privacy thresholds).
Align order / sale pipeline so confirmed orders flow into sales (or a new order_analytics mart) for consistency.
5. Premium listing
What you have today

No first-class paid placement or sponsored rank in schema or compare logic.
pharmacies.chain exists; ranking is price and distance (fair, not monetized).
Gaps

Entire monetization surface: billing, caps, disclosure (“Ad”), A/B on rank.
Suggested approach

Product: listing_tier or sponsored_until on pharmacies / pharmacy_prices; auction or fixed fee per city+SKU or per pharmacy+zone.
Ranker: score = organic_score + bid_weight * sponsored_score with hard caps so paid never fully buries cheaper options (regulatory/trust).
Transparency: label in UI; per-request audit of rank components for partners.
Start MVP: fixed “featured pharmacy” slots in a PIN/micro-market before full auction.
